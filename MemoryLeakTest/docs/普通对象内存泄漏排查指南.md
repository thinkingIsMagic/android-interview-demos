# 普通对象内存泄漏排查指南

> 排查非 Activity 类型的普通对象（如 `List`、`Map`、自定义 `Manager` 或 `Thread`）内存泄漏，其实比排查 Activity 更有挑战性，因为它们没有明显的生命周期回调。

不过别担心，核心逻辑依然是：**找到谁在不该持有它的时候，依然拽着它不放。**

---

## 1. 定位工具：三板斧

对于非 Activity 对象，手动代码检查效率太低，必须依靠工具观察堆内存（Heap）。

### 1.1 LeakCanary (首选)

虽然它默认监测 Activity，但你可以通过 `AppWatcher.objectWatcher.watch(myObject)` 手动观察任何对象。

```kotlin
import leakcanary.AppWatcher

// 手动观察任意对象
AppWatcher.objectWatcher.watch(myObject, "自定义描述")
```

### 1.2 Android Studio Profiler

实时查看内存曲线。如果内存只增不减，点击 **Capture Heap Dump**。

### 1.3 MAT (Memory Analyzer Tool)

分析 `.hprof` 文件的终极工具。虽然 UI 像上个世纪的，但功能极其强大。

---

## 2. 核心排查步骤

### 第一步：获取堆转储文件 (Heap Dump)

在应用运行一段时间并执行了疑似导致泄漏的操作后，使用 Profiler 捕获堆快照。

### 第二步：利用 "Path to GC Root"

这是排查普通对象的**关键**。在 Profiler 或 MAT 中：

1. 找到你的目标类（按类名搜索）
2. 查看该类的实例（Instance）
3. 右键选择 **"Show Nearest GC Root"** (排除掉软引用/弱引用)

> **原理解析：** 一个对象之所以不被回收，是因为它到 **GC Root**（如运行中的线程、静态变量、JNI 引用）之间存在一条引用链。

### 第三步：识别常见的"幕后黑手"

如果不是 Activity，通常泄漏发生在以下几个地方：

| 泄漏原因 | 典型特征 | 排查方法 |
|---------|---------|---------|
| **静态集合** | `static List/Map` 持续增长 | 查看集合大小是否无限增加 |
| **未关闭的线程/定时器** | 后台线程仍在运行 | 检查线程状态 |
| **单例模式** | 单例持有 Activity 上下文 | 检查单例中的引用 |
| **未注销的监听器** | 回调持续触发 | 检查注册/注销配对 |

---

## 3. 分析技巧：对比大法

如果你不确定哪个对象泄露了，可以使用 **"对比快照"**：

1. **快照 A：** 应用刚启动时的状态
2. **操作：** 反复执行某个功能
3. **快照 B：** 操作完后的状态
4. **对比：** 查看在快照 B 中，哪些普通对象的实例数量（Instance Count）比快照 A 多且没有回落

## 4. 常见代码场景修复

### 4.1 静态集合泄漏

```kotlin
// 泄漏
object StaticCacheManager {
    val leakyCache = mutableListOf<Any>()
    fun add(data: Any) = leakyCache.add(data)
}

// 修复：限制大小或使用 LruCache
object FixedCacheManager {
    private val cache = LruCache<String, Any>(100)
    fun add(key: String, data: Any) = cache.put(key, data)
    fun clear() = cache.evictAll()
}
```

### 4.2 线程/定时器泄漏

```kotlin
// 泄漏：匿名内部类持有外部类引用
class NetworkManager {
    private val thread = Thread {
        // 持有 NetworkManager 引用
    }
}

// 修复：使用静态内部类 + WeakReference
class SafeNetworkManager {
    private var thread: Thread? = null
    fun startWork() {
        thread = Thread(SafeRunnable(WeakReference(this)))
        thread?.start()
    }

    private class SafeRunnable(
        private val managerRef: WeakReference<SafeNetworkManager>
    ) : Runnable {
        override fun run() {
            managerRef.get()?.doWork()
        }
    }
}
```

### 4.3 单例模式泄漏

```kotlin
// 泄漏：持有长生命周期引用
object LeakySingleton {
    private var context: Context? = null
    fun setContext(ctx: Context) { context = ctx }
}

// 修复：使用弱引用
object SafeSingleton {
    private var contextRef: WeakReference<Context>? = null
    fun setContext(ctx: Context) { contextRef = WeakReference(ctx) }
}
```

### 4.4 监听器未注销

```kotlin
// 泄漏：只注册不注销
class LocationTracker {
    private val listeners = mutableListOf<LocationListener>()
    fun register(listener: LocationListener) { listeners.add(listener) }
    // 缺少 unregister
}

// 修复：确保配对注销
class SafeLocationTracker {
    private val listeners = mutableListOf<LocationListener>()
    fun register(listener: LocationListener) { listeners.add(listener) }
    fun unregister(listener: LocationListener) { listeners.remove(listener) }
    fun onDestroy() { listeners.clear() }
}
```

### 4.5 Handler 消息未处理

```kotlin
// 泄漏
class MessageProcessor {
    private val handler = Handler(Looper.getMainLooper()) {
        // 持有 MessageProcessor 引用
    }
    fun sendDelayedMessage() {
        handler.sendMessageDelayed(msg, 60000)
    }
}

// 修复：在 onDestroy 中清理
class SafeMessageProcessor {
    private val handler = Handler(Looper.getMainLooper()) { ... }
    fun cleanup() {
        handler.removeCallbacksAndMessages(null)
    }
}
```

## 5. 总结建议

如果你已经拿到了堆转储文件（Heap Dump），可以重点关注 **"Objects with high retained size"**（深堆占用最高的实例）。这通常能直接指出是哪个 Map 或 List 吞掉了内存。

## 6. 演示项目使用说明

本项目提供了以下演示场景：

1. **静态集合泄漏** - 点击"静态集合缓存泄漏"按钮，模拟数据持续添加到静态 List
2. **线程/定时器泄漏** - 启动后台线程和定时任务，观察线程持有的引用
3. **单例模式泄漏** - 将 Activity 传递给单例，模拟生命周期不匹配导致的泄漏
4. **监听器泄漏** - 注册监听器但不注销

每个场景都配合了 LeakCanary 手动观察功能，可以在 Logcat 或通知中查看泄漏分析结果。
